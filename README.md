# 💫 About Me:
🔭 I’m currently working on<br>At Akquinet, I’m basically living in the “ports + simulation + AI” world right now. I’m building digital twin simulations for port terminal operations so we can test different scenarios without touching live systems—things like how throughput changes, where delays pile up, and what causes queues to explode during busy periods. A big part of my day is taking a real operational question (like “what if the yard is congested?” or “what if a navigation-related failure happens?”), translating it into a simulation setup, running experiments, and then pulling out metrics that actually make sense to operators.  ￼<br><br>On top of the simulation side, I’m also working on LLM-powered decision support for port operations. The focus is not “chatbot vibes” — it’s more like: the operator asks something, and the system responds with policy-compliant, traceable recommendations using LangChain + RAG, so the output isn’t random and can be justified when someone asks “why are we doing this?”.  ￼<br><br>And because this is meant to be demoable and eventually deployable, I’m also doing the “real engineering” glue: containerizing simulation components with Docker / Docker Compose, writing technical documentation that other people can actually follow, and setting up CI/CD-style workflows so stakeholder demos don’t become a last-minute panic.  ￼<br><br>👯 I’m looking to collaborate on<br>Anything in the space of digital twins / simulation-based evaluation, especially if it connects to operations metrics (throughput, delay, queueing) and “what-if” analysis. I’m also very into RAG systems that behave like a product: grounded answers, citations/traceability, and tight feedback loops so the system improves instead of drifting.  ￼<br><br>🤝 I’m looking for help with<br>Honestly: making these systems more “industry-solid.” Stuff like better ways to structure simulation experiments so results are repeatable, plus robust evaluation of RAG outputs (grounding/citation accuracy), and production hardening basics (deployment patterns, monitoring, CI/CD discipline).  ￼<br><br>🌱 I’m currently learning<br>More of the “ship it properly” side: Docker-to-Kubernetes mindset, cleaner CI/CD habits, and how to design decision-support outputs that are not only correct, but also auditable and operator-friendly (because ports don’t care if your model is fancy if it can’t be trusted).  ￼<br><br>💬 Ask me about<br>Port terminal simulation experiments, turning messy operational questions into measurable scenarios, building LangChain + RAG systems that don’t hallucinate their way into trouble, and packaging everything into containers so it can run reliably for demos and iteration.  ￼<br><br>⚡ Fun fact<br>I like work that turns “signals” into something usable — whether it’s operational metrics from simulations or security signals into dashboards/pipelines. If there’s a messy system, I’m usually the person trying to make it measurable and automatable.  ￼
